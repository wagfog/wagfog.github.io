最简单的操作系统

我们有了NEMU+AM+IOE就可以把正小游戏放在上面运行啦

但是会独占整个操作系统

这样如果要执行下一个程序就需要管理员来手动加载新的指令，这太麻烦了

回想起冯诺依曼计算机的工作方式, 它的其中一个特点就是: 当计算机执行完一条指令的时候, 就自动执行下一条指令. 类似的, 我们能不能让管理员事先准备好一组程序, 让计算机执行完一个程序之后, 就自动执行下一个程序呢?

这就是批处理系统的思想:

有前台程序也有后台程序，当前台程序结束时后台程序就加载一个新的前台程序

我们称这个后台程序为：操作系统

是的，操作系统也不过是一个C程序罢了！

我们将带着这样的疑问去探索操作系统：操作系统和普通的C程序有什么不同之处呢？、

回顾历史, 要实现一个最简单的操作系统, 就要实现以下两点功能:

  • 用户程序执行结束之后, 可以跳转到操作系统的代码继续执行
  • 操作系统可以加载一个新的用户程序来执行

来自操作系统的新需求

当然，我们首先会想到jal/call指令

但是：个用户程序出错了, 操作系统可以运行下一个用户程序; 但如果操作系统崩溃了, 整个计算机系统都将无法工作

所以操作系统还是应该被保护起来的

riscv32

riscv32提供 ecall指令作为自陷指令, 并提供一个stvec寄存器来存放异常入口地址. 为了保存程序当前的状态, riscv32提供了一些特殊的系统寄存器, 叫控制状态寄存器(CSR寄存器). 在PA中, 我们只使用如下3个CSR寄存器:

  • sepc寄存器 - 存放触发异常的PC
  • sstatus寄存器 - 存放处理器的状态
  • scause寄存器 - 存放触发异常的原因
  • riscv32通过 sret指令从异常处理过程中返回, 它将根据sepc寄存器恢复PC.

将上下文管理抽象成CTE

我们刚才提到了程序的状态, 在操作系统中有一个等价的术语, 叫”上下文”

硬件提供的上述在操作系统和用户程序之间切换执行流的功能, 在操作系统看来, 都可以划入上下文管理的一部分.

而上下文的具体内容, 在不同的架构上也显然不一样(比如寄存器就已经不一样了). 于是, 我们可以将上下文管理的功能划入到AM的一类新的API中, 名字叫CTE(ConText Extension).

  • 如何做到呢？
  • 先思考操作系统的处理过程其实需要哪些信息
    • 引发执行流切换的原因，除以0还是非法指令还是出发断点，还是程序自愿陷入操作系统?
    • 程序的上下文，操作系统读出上下文中的一些寄存器，根据它们的信息啦自己西宁进一步的处理

把这两点信息抽象成一中统一的表达方式，就可以定义出CTE的API了

CTE定义了名为“事件”的如下数据结构

event表示事件编号, cause和 ref是一些描述事件的补充信息

msg是事件信息字符串,

具体内容, 就无法进一步进行抽象了. 这主要是因为不同架构之间上下文信息的差异过大

最后还有另外两个统一的API:

  • bool cte_init(Context* (*handler)(Event ev, Context *ctx))用于进行CTE相关的初始化操作. 其中它还接受一个来自操作系统的事件处理回调函数的指针, 当发生事件时, CTE将会把事件和相关的上下文作为参数, 来调用这个回调函数, 交由操作系统进行后续处理.
  • void yield()用于进行自陷操作, 会触发一个编号为 EVENT_YIELD事件. 不同的ISA会使用不同的自陷指令来触发自陷操作, 具体实现请RTFSC.

可执行文件如何组织

为了回答剩下的问题, 我们首先需要了解可执行文件是如何组织的. 你应该已经在课堂上学习过ELF文件格式了, 它除了包含程序本身的代码和静态数据之外, 还包括一些用来描述它们的信息, 否则我们连代码和数据之间的分界线在哪里都不知道. 这些信息描述了可执行文件的组织形式, 不同组织形式形成了不同格式的可执行文件, 例如Windows主流的可执行文件是PE(Portable Executable)格式, 而GNU/Linux主要使用ELF(Executable and Linkable Format)格式.

如何是识别不同格式的可执行文件？

GNU/Linux尝试加载一个可执行文件时，会首先检查文件的魔数（magic Number），这是一个特性的字节徐铁，如果检测到的魔数与系统期望的ELF格式不符合，就会报告“格式错误”

ELF提供的两个视角来组织一个可执行文件

  • 面向链接过程的section视角
    • 提供用于链接和重定位的信息(符号表)
  • 面向执行的segment视角
    • 提供用于加载可执行文件的信息
  • 一个segment可能由0个或多个section组成, 但一个section可能不被包含于任何segment中.

我们现在侧重于去看加载这一方面的内容

ELF中采用program header table来管理segment, program header table的一个表项描述了一个segment的所有属性

  • 类型
  • 虚拟地址
  • 标志
  • 对齐方式
  • 文件内偏移量和segment的大小

加载一个可执行文件并不是加载塔的所包含的所有内容，只要加载那些与运行时刻相关的内容就可以了

我们可以通过判断segment的 Type属性是否为 PT_LOAD来判断一个segment是否需要加载.

堆和栈在哪

为什么堆和栈的内容没有放到可执行文件里？

  • 栈
    • 栈是由编译器自动分配的内存
  • 堆

冗余的属性

一个segment包含两个大小的属性, 分别是 FileSiz和 MemSiz, 这是为什么? 再仔细观察一下, 你会发现 FileSiz通常不会大于相应的 MemSiz, 这又是为什么?

  1. FileSiz ：FileSiz指定了该段在可执行文件中占用的大小。这个大小是指这个段在磁盘上占用的字节数。它包括了段所包含的数据和元数据的大小。
  2. MemSiz ：MemSiz指定了该段在运行时所占用的大小。这个大小是指这个段在内存中占用的字节数。它包括了该段所包含的数据以及需要在运行时进行动态分配的空间。

通常情况下，FileSiz不会大于相应的MemSiz，这是由于以下原因：

  • 对于只读段（如代码段） ：可执行文件中的代码段是只读的，并且在运行时不允许对其进行修改。因此，FileSiz和MemSiz的大小通常是相等的，以确保代码段在内存中分配的大小与文件中的大小一致。
  • 对于可写段（如数据段） ：可执行文件中的数据段包含了在程序运行时需要进行修改的数据。这可能包括全局变量、静态变量等。在这种情况下，FileSiz指定了段在文件中的大小，而MemSiz指定了在运行时为此段分配的内存大小。由于在运行时可能需要进行动态分配，因此MemSiz可能会大于FileSiz。

为什么清零

将 [VirtAddr + FileSiz, VirtAddr + MemSiz) 对应的物理区间清零是为了：这是暂时未用到的内存，清零可以防止未知错误

系统调用与运行时环境

系统调用把整个运行时环境分成两部分

  • 操作系统内核区
    • 访问系统资源的功能
  • 用户区
    • 保留无需使用系统资源的功能
      • 比如标准库

这样，用户程序只能用于计算，超越纯粹计算能力之外的任务都需要通过系统调用向操作系统请求服务

系统调用

  • 如何出发一个系统调用
    • 通过自陷操作可以告诉操作系统相关问题
  • 系统调用函数只需要在上下文中获取必要信息，就知道用户程序发出的请求是什么了
    • 既然我们通过自陷指令来触发系统调用, 那么对用户程序来说, 用来向操作系统描述需求的最方便手段就是使用通用寄存器了, 因为执行自陷指令之后, 执行流就会马上切换到事先设置好的入口, 通用寄存器也会作为上下文的一部分被保存起来
  • Nanos-lite收到系统调用事件之后, 就会调出系统调用处理函数 do_syscall()进行处理
    • 然后根据上下文内容对不同的事件进行分发
  • 然后设置系统调用的返回值，放在指定的寄存器中
  • 经过CTE, 执行流会从 do_syscall()一路返回到用户程序的 _syscall_()函数中. 代码最后会从相应寄存器中取出系统调用的返回值, 并返回给 _syscall_()的调用者, 告知其系统调用执行的情况(如是否成功等).

堆区管理

调整堆区大小是通过 sbrk()库函数来实现的, 它的原型是

用于将用户程序的program break增长 increment字节,

在GNU/Linux中, 这个系统调用是 SYS_brk, 它接收一个参数 addr, 用于指示新的program break的位置. _sbrk()通过记录的方式来对用户程序的program break位置进行管理, 其工作方式如下:

  1. program break一开始的位置位于 _end
  2. 被调用时, 根据记录的program break位置和参数 increment, 计算出新program break
  3. 通过 SYS_brk系统调用来让操作系统设置新program break
  4. 若 SYS_brk系统调用成功, 该系统调用会返回 0, 此时更新之前记录的program break的位置, 并将旧program break的位置作为 _sbrk()的返回值返回
  5. 若该系统调用失败, _sbrk()会返回 -1

上述代码是在用户层的库函数中实现的, 我们还需要在Nanos-lite中实现 SYS_brk的功能. 由于目前Nanos-lite还是一个单任务操作系统, 空闲的内存都可以让用户程序自由使用, 因此我们只需要让 SYS_brk系统调用总是返回 0即可

  • 究竟是怎么执行到它的第一条指令的?
    • 加载器最后返回了一个这指向指令在内存中的虚拟地址的指针
  • hello程序在不断地打印字符串, 每一个字符又是经历了什么才会最终出现在终端上?
    • 通过printf来进行打印
    • printf函数先是通过调用sbrk来对堆区进行分配，由此得到了缓冲区
    • 然后再把所有的要输出的内容放到缓冲区当中
    • 再通过系统调用sys_wirte来把缓冲区的内容一次型输出出来
      • 原来是这样！！！！这就解开了我一直以来对对于关于IO操作的缓冲区的不理解！

简易文件系统

  • 对于用户程序来说，怎么知道文件在ramdisk中的什么位置？
    • 操作系统需要在存储截止的驱动程序之上为用户程序提供一个更高级的抽象：文件
  • 文件的本质：字节序列 + 一些额外属性

在我们的文件系统中，：

  • 文件大小固定
  • 写文件时不允许超过原有文件的大小
  • 文件的数量是固定的
  • 不能创建新文件
  • 没有目录
    • 所以我们把目录分隔符以及应该存在的目录层级都直接写在文件名字中

这样设计的由来

众所周知，计算机会连接许多设备，如果用户程序想要访问设备，怎么办呢？

还是直接syscall吗？

当然可以，但是问题也是有的

直接系统调用访问设备的问题

  • 设备的类型非常多，每个类型还有许多数不胜数的功能
    • 如果给每个功能都配备系统调用接口，那简直太夸张了
  • 设备的功能差别大，接口难以统一
    • 接口不能题意的后果就是程序与设备之间的交互会变得困难

好处

计算机中到处都是字节序列(比如：内存，管道，磁盘，socke，甚至一些非存储类型的硬件例如键盘敲下的按键编码形成了一个字节序列)，文件的本质也是字节序列

那么，利用文件作为接口来和设备进行交互，就可以统一接口了！

终极拷问

当你在终端键入 ./hello运行Hello World程序的时候, 计算机究竟做了些什么?

  1. 设置当前目录为暂时的环境变量
  2. 调用execvp(“hellp”,argv);函数发出系统调用申请来执行程序
  3. 发出中断指令，触发exec系统调用
  4. 把当前寄存器中和内存中的数据保存到栈或者某段内存中
  5. 控制权来到操作系统内核，根据调用类型执行相应的操作和功能
  6. 接下来就是执行程序，CPU从加载器加载好的内存中一条条读取指令
  7. 在碰到需要输入输出的时候，触发又一次系统调用，wrtie系统调用，系统调用操作同上
  8. 在堆区分配好缓存区域，把要输出到屏幕上的内容全部放到缓存区域
  9. 然后把缓存中内容一次性输出到屏幕上
